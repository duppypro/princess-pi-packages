Research: /serve Refactor — Static Complexity Analysis

Domain: Code Quality, Static Analysis, Maintainability Measurement

1. What Changed

A maintainability review of extensions/serve.ts identified a single 237–284 line command handler containing a sequential if-chain for all six /serve subcommands, plus duplicated process-kill logic. The refactor (commit e0cbefe, parent 386a74a) extracted six named handler functions (handleLog, handleHelp, handleHide, handleShow, handleKill, handleStart) plus a small dispatch table, and added a shared killProcess(pid) helper in extensions/lib/serve/process.ts. Behavior was verified unchanged via direct execution against a mocked pi/ctx harness covering all 7 routes.

This report measures whether that refactor produced a measurable improvement, using static analysis tools rather than impression alone.

2. Methodology

  • Before: git show 386a74a:extensions/serve.ts (the commit immediately prior to the refactor).
  • After: extensions/serve.ts at HEAD (commit e0cbefe).
  • Type stripping: both versions transpiled TS→JS via esbuild --format=cjs --target=es2015, since the complexity tools below predate TypeScript and can't parse type annotations/interfaces. Lowering to ES2015 also converts async/await into generator-based state machines — relevant when reading the raw numbers below (see §5 caveats).
  • Tool 1 — plato v1.7.0 (wraps escomplex): cyclomatic complexity, Halstead metrics (Volume, Difficulty, Effort, estimated Bugs), Maintainability Index, per-function breakdown.
  • Tool 2 — ESLint complexity rule (built-in, threshold set to 1 to report every function's score): independent cyclomatic complexity calculation, used to cross-validate Tool 1's numbers rather than trust a single source.

Halstead Volume is the closest thing to an information-theoretic "entropy" measure in this toolset: V = N · log₂(n), where n is the vocabulary size (distinct operators + operands) and N is the program length (total operators + operands). It measures the size of the information content a reader must process — not how well that information is organized.

Raw Δ is always the literal change in the number (so "+" always means the value increased). The Verdict column is the separate judgment of whether that increase is good or bad for that specific metric — they are not the same axis, which is why an increasing metric can still be marked unfavorable.

Taken alone, this table looks unflattering — most raw totals went up slightly. That's expected and not a red flag: total behavior didn't change, so total decision-logic and vocabulary couldn't shrink. What moved in the right direction is the Maintainability Index, which is the one metric here explicitly designed to reward spreading logic across more, smaller units rather than just summing raw totals.

5. Named, Independently-Readable Units

BeforeAfter
getOrCreateCertificates (only named helper besides the wrapper)getOrCreateCertificates, handleLog, handleHelp, handleHide, handleShow, handleKill, handleStart

Every subcommand now has a name, a defined boundary, and could be unit-tested in isolation by calling it directly with a mocked ctx — which is exactly how this refactor was verified (see commit e0cbefe).

Verdict: the refactor did not reduce the total amount of logic in the file (it couldn't, without changing behavior) — it redistributed that logic from one 237–284 line, complexity-34/42 blob into 6 named functions, the worst of which now sits at complexity 14/15 and 93 lines. The Maintainability Index improvement (+10.5%) and the worst-case reduction (~60–67% across complexity, volume, and estimated-bugs) are the two metrics that actually capture this; the flat-to-slightly-worse aggregate totals are an artifact of measuring total information content rather than its organization, and should not be read as the refactor having no effect.
Caveats:
  • These tools (escomplex, last published ~2016) predate TypeScript and can't parse it directly — both versions were transpiled to plain JS via esbuild first. Absolute numbers reflect the transpiled output, not the original TS source line-for-line, though the comparison is fair since both sides went through an identical pipeline.
  • Lowering to ES2015 target converts async/await into generator-based state machines, which is why ESLint reports the top function as a "Generator function" rather than an arrow function, and partially explains why its complexity count (42) differs from escomplex's (34) for the same logical unit.
  • Cyclomatic complexity, Halstead metrics, and the Maintainability Index are all decades-old heuristics with known blind spots (e.g. they can't tell well-named code from poorly-named code, or judge whether closures-over-shared-state are the right design choice for a given file). They're a useful cross-check against impression, not a replacement for the human review that originally motivated this refactor.

6. Reproduction

mkdir -p /tmp/serve-analysis
git show 386a74a:extensions/serve.ts > /tmp/serve-analysis/before.ts
cp extensions/serve.ts /tmp/serve-analysis/after.ts
cd /tmp/serve-analysis
npx --yes esbuild before.ts --format=cjs --target=es2015 --outfile=before.js
npx --yes esbuild after.ts  --format=cjs --target=es2015 --outfile=after.js
npx --yes plato -r -d plato-before before.js
npx --yes plato -r -d plato-after  after.js
# per-function detail: plato-before/files/before_js/report.json (same for after)
# ESLint cross-check: eslint --no-config-lookup -c eslint.config.mjs {before,after}.js
#   with rule: complexity: ["warn", 1]
