Serve Utility (EXT_SERVE.html)

Domain: Local Development, Dev Server & Hot-Reloading

Overview

The Serve utility provides a custom, native, zero-dependency Node.js development server. It implements instant Server-Sent Events (SSE) live-reloading, automatic cache-busting, and a dynamic HTML/JS/CSS dependency-tracing engine. It also provides a --static fallback option to utilize standard http-server behavior when live-reloading is not desired.

Core Architecture

1. Native HTTPS File Server (Default Mode)

  • Zero Dependencies: Built strictly using Node.js standard library modules (node:https, node:fs, node:path, node:mime/custom mappings).
  • Cache Control: Sets headers to strictly bypass browser caches (Cache-Control: no-store, no-cache, must-revalidate) to prevent sticky dev states.
  • Client Script Injection: Automatically intercepts outgoing .html file read streams to inject a tiny (~15 lines) native SSE listener script before the closing </body> tag.

2. Lightweight Live-Reload Protocol (SSE)

  • Server-Sent Events: Avoids the overhead of external WebSockets packages (like ws). Exposes a special endpoint (/__live-reload) using standard HTTP/HTTPS protocols.
  • Auto-Reconnections: Browsers natively support reconnecting to SSE event streams out-of-the-box via the built-in EventSource API.

3. FS Watcher & Debounce Queue

  • Native Watching: Uses Node's native fs.watch (recursive) to monitor directory modifications.
  • Debounce Window: Groups multiple rapid file changes (such as auto-formatter writes or multi-file saves) into a single batched event, delaying broadcasts by a configurable cooldown window (default: 150ms).

4. Dynamic Dependency-Tracing Engine

To keep reloads highly efficient and prevent unnecessary cycles, the server builds an internal in-memory Dependency Graph on startup and updates it dynamically on file saves:

  • HTML Scans: Scans `.html` files for <link rel="stylesheet">, <script src="...">, and fetch(...) endpoints to map stylesheets, scripts, and JSON configs to their HTML pages.
  • JS Scans: Scans native ES modules for static and dynamic import statements (import ... from '...').
  • Graph Lookup & Hot-Swapping:
    • If a CSS file changes, the server triggers a targeted { type: 'css', path: '/css/style.css' } payload. The injected client script updates the DOM link reference with a cache-busting timestamp on-the-fly, giving instant visual updates with zero page reloads.
    • If a JS or HTML file changes, the server traces up the dependency graph. If the file is imported by the currently viewed page, it triggers a page reload ({ type: 'reload' }); if the changed file is unused, it suppresses the reload entirely.

5. Static Fallback (--static / -s)

If a user desires to run a directory without any code-injection, cache-busting, or live-reload watchers, passing the --static flag will bypass the native Node.js Live Server and instead spawn a detached npx http-server instance directly over the targeted directory.

Step-by-Step Data Flow

  1. User executes /serve.
  2. Server reads persistent keys from ~/.pi-certs/ and spins up the native HTTPS Server.
  3. Server traverses the target workspace directory to build the dependency map.
  4. Browser requests does-it-glider.html.
  5. Server reads the file, injects the SSE Client Script, and streams it to the browser.
  6. Browser establishes connection with /__live-reload.
  7. Developer modifies game-state.js.
  8. FS Watcher detects the write, debounces it, checks the dependency map to confirm it is imported by the active document, and pushes a { type: 'reload' } payload down the SSE connection.
  9. Browser refreshes instantly.

The Injected SSE Client Script

A minimal, vanilla injection script appended to outgoing HTML streams:

<script>
  (function() {
    const es = new EventSource('/__live-reload');
    es.onmessage = (e) => {
      const data = JSON.parse(e.data);
      if (data.type === 'css') {
        const links = document.querySelectorAll('link[rel="stylesheet"]');
        for (const link of links) {
          const url = new URL(link.href, window.location.href);
          if (url.pathname === data.path || url.pathname.endsWith(data.path)) {
            link.href = url.pathname + '?t=' + Date.now();
            break;
          }
        }
      } else if (data.type === 'reload') {
        location.reload();
      }
    };
  })();
</script>

CLI Mode (Outside Pi Coding Agent) — Cross-Harness

bin/serve.mjs is a standalone, Pi-independent port that provides zero-token command execution outside of Pi. Just like wtft and merge, it uses esbuild to natively bundle its shared TypeScript logic (from extensions/lib/serve/) into a single, pure ESM JavaScript binary. Because it is pure JavaScript (#!/usr/bin/env node), it can be installed globally for cross-harness use without triggering Node's --experimental-strip-types node_modules errors.

Installation (Global CLI):

npm install -g github:duppypro/princess-pi-packages

Once installed, invoke it as the npm-global serve [args], via the ./serve wrapper from the repo root, or through Claude Code's shell-escape prefix (e.g. !serve --log). Since Claude Code has no extension-dispatch mechanism to bypass the LLM reasoning loop like Pi's registerCommand, this CLI enables identical zero-token usage in that harness.

  • Supported: default start, --static/-s, --log/-L, --kill/-k, --help/-h — identical behavior to the Pi extension.
  • Omitted: --hide/-H and --show/-S have no CLI equivalent — there's no persistent TUI widget outside Pi to hide or show.
  • Added: --force/-f bypasses the .env secret warning. The Pi extension can block on an interactive confirm dialog; the CLI can't safely assume a TTY is attached, so it skips directories containing .env by default and requires this flag to proceed.

Bounded Context & Rules

  • Stateless Tracking: Servers are identified dynamically by scanning the host OS. A server belongs to the active context if its served directory resides within process.cwd().
  • Security: Scans target directories for .env files and prompts for confirmation before exposing them over the network.
  • Deduplication: Prevents multiple HTTP servers from spawning on the exact same absolute directory path.
  • Dual Dev Modes: Runs as a native, live-reloading server with hot CSS injections by default. Passing the --static or -s flag falls back to a standard static HTTP/HTTPS file server (spawning http-server) with no code injections or watchers.

Security & Certificates

  • Certificate Storage: SSL certificates (cert.pem and key.pem) are stored persistently in the user's home directory under ~/.pi-certs/. This ensures they survive system reboots and prevents browser certificate fingerprint mismatch warnings.
  • Auto-Generation: If missing, 2048-bit RSA self-signed certificates are automatically generated securely using the host's openssl CLI.
  • Conflict Prevention: Storing certificates in the user's home directory (with owner-only read/write permissions) avoids the permission conflicts and security risks associated with shared temporary directories.
  • Access Logging & Automatic Rotation: The custom dev server natively implements Apache Common Log Format access logging. Logs are written to ~/.pi-certs/logs/port-<PORT>-access.log. To prevent logs from consuming excessive disk space, the server includes a built-in, cross-platform, zero-dependency log rotation function. Once an active log exceeds 5MB, it is automatically rotated to a .old.log backup (capping total disk usage per server at exactly 10MB). This operates entirely inside the background process independent of any active Pi CLI sessions.
  • VPS Firewall Protection: Because /serve exposes directories to all network interfaces, users on a VPS must explicitly restrict port access using their host's firewall (e.g., UFW or Hostinger Control Panel) to block unauthorized traffic.

Worktree Resolution & Multi-Serving

By relying on the active session's directory (ctx.cwd) rather than a fixed process directory, the Serve utility dynamically tracks your active worktree. Note that Unix process limitations prevent child processes (such as sub-shells spawned via !cd) from modifying the parent Node process's working directory. Therefore, to switch your active session's worktree, you must quit and resume your session from the host terminal.

Dynamic Worktree Categorization

The TUI widget dynamically filters and groups all active servers on the system into two distinct categories:

  • This Worktree: Active servers whose root directory matches or resides within the active session's directory (ctx.cwd).
  • Other: All other active servers on the host machine.
  • The 'all' Target: Running /serve --kill all (or -k all) sweeps the host system and immediately terminates all active development servers (both live and static) in both the current worktree and all other folders.

Step-by-Step Multi-Worktree Serving Example

To serve multiple independent projects simultaneously:

# 1. Start Pi in your first worktree and serve it:
$ cd /home/princess-pi/git-projects/does-it-glider-juice-my-extensions
$ pi
/serve

# 2. Pause the Pi session:
/quit

# 3. Change directory on the host terminal and resume your session in the second worktree:
$ cd ../does-it-glider-simple-prompt
$ pi -r

# 4. Serve the second project (will automatically choose next available port, e.g., 8082):
/serve

Settings Persistence (Cross-Harness Config)

Serve uses the shared extensions/lib/config.ts module for harness-agnostic settings persistence. Config files live at ~/.config/princess-pi/serve.json (user-global) and ./.princess-pi/serve.json (project-local, overrides global). CLI flags take highest precedence.
