EXT_DEDUP: Streaming Log Deduplicator

A shell pipe filter and Pi extension. Primary use: tail -f app.log | dedup. Compresses repeated adjacent log lines into a single annotated summary, tracking number ranges and sequence direction across duplicate groups.

Verdict: Make. No existing tool combines streaming, in-place terminal rewriting, and number-range analysis. The closest is Chrome DevTools console collapsing, but that's browser-only and lacks numeric intelligence. uniq -c is the only real competition for the basic case but it can't rewrite in place and emits counts only after a group closes.

Step 2 — Feature Spec

2.1 Core Algorithm

  1. Read stdin line-by-line.
  2. For each line, compute its template: replace all numbers (integers, floats, negative) with the placeholder {N}.
  3. If the template matches the current group's template → it's a duplicate: increment count, accumulate the number values in per-slot lists.
  4. If the template does not match → group ends. Emit summary (see §2.2), start new group.
  5. On EOF → flush the final group.

2.2 Output Modes

TTY mode (stdout is a terminal, e.g. tail -f app.log | dedup):

  • First line of a group: print normally.
  • On each duplicate: use ANSI escape codes to move cursor up one line, clear it, and reprint as 👆 ×N [reconstructed line].

Non-TTY mode (piped output, e.g. tail -f app.log | dedup | grep foo):

  • First line of a group: print immediately.
  • Duplicates: silent (accumulate only).
  • When group ends with count > 1: print ↑ ×N [reconstructed line] as a follow-on annotation line.

2.3 Number Range and Sequence Display

When reconstructing the summary line, each {N} slot is replaced with a formatted summary of all values observed in that slot:

  • Constant (all same value): show the value as-is — 5432
  • Contiguous integer range: [3-7]
  • Gapped ranges: [3-5, 8-10]
  • Non-contiguous integers: [1, 3, 7]
  • Floats: [1.2-3.7] (min–max only)

2.4 Sequence Direction Annotation

Appended to the range when the values form a detectable pattern (requires ≥ 3 values):

  • Uniform step up: ↑2 (e.g. values 2, 4, 6, 8)
  • Uniform step down: ↓2
  • Irregular increase: ↑
  • Irregular decrease: ↓
  • Fluctuating: no annotation

Step 4 — I/O Strategy: readline vs Chunk-Buffered stdin

After the initial implementation shipped (readline-based), a parallel agent working the same GitHub issue under the name yada claimed a 32.7x speedup by replacing Node's readline module with manual chunk-buffered stdin.on("data") parsing. That approach was ported here and benchmarked directly against the original.

4.1 What Changed

  • extensions/lib/dedup/io-readline.ts — the original implementation, extracted unchanged. Kept only as the benchmark baseline.
  • extensions/lib/dedup/io-chunked.ts — ported from yada's approach: reads raw stdin buffers via "data" events, manually splitting on \n with a leftover-buffer carry, instead of going through readline.createInterface.
  • extensions/dedup.ts now calls runFilterChunked() as the live entry point. Both functions accept injectable Readable/Writable streams (defaulting to process.stdin/process.stdout) so they can be driven in-process for benchmarking, without spawning real subprocesses.
  • extensions/lib/dedup/benchmark.ts — in-process benchmark comparing both, against a synthetic 17,000-line fixture (bursts of near-duplicate lines separated by unique lines, chunked into ~64KB pieces to mimic real OS pipe behavior).

4.2 Benchmark Results

ImplementationAvgSpeedup
readline (baseline)~30–35ms1.00x
chunk-buffered~28–34ms1.01–1.05x

At 300,000 lines the ratio holds at ~1.03x — it does not widen with scale, which rules out an O(N²) vs O(N) explanation for the difference.

The 32.7x claim does not hold up. Two things were checked:
  1. yada's own benchmark.ts spawns a fresh subprocess per timed iteration and references bin/yada-readline.ts as the baseline — a file that was never committed to that repo. Running it directly throws MODULE_NOT_FOUND. The 32.7x number is not reproducible from what's actually in that repo.
  2. yada's real complexity fix, found by diffing their two relevant commits, was a one-line guard (if (isTTY && !isFastMode) renderBlock(...)) that stops an expensive per-line periodicity/range recomputation from running on every duplicate line in batch mode — an O(N²) → O(N) fix entirely orthogonal to readline vs chunked stdin parsing. This codebase's buildSummaryLine was never called redundantly per line in the first place, so there was no equivalent bug to fix here.
Chunk-buffered stdin is still adopted as the live implementation since it's a measured, if modest, real win with no downside — but the spec record should not imply it was ever an O(N²)-class fix.

Implementation Notes

  • File layout: Pure logic in extensions/lib/dedup/core.ts; stream I/O strategies in extensions/lib/dedup/io-chunked.ts (live) and io-readline.ts (benchmark baseline); Pi extension + standalone entry in extensions/dedup.ts; tests in extensions/lib/dedup/core.test.ts; benchmark in extensions/lib/dedup/benchmark.ts.
  • Standalone run: npx tsx extensions/dedup.ts (detects entry point via import.meta.url).
  • Benchmark run: npx tsx extensions/lib/dedup/benchmark.ts — drives both I/O strategies in-process against a synthetic fixture, avoiding subprocess-spawn overhead in the measurement.
  • Integer contiguity: Range gaps are computed on integer values only. Floats get min–max display.
  • Sequence threshold: Direction/step annotation requires ≥ 3 values to be reliable.
  • Terminal width: In-place ANSI rewrite assumes the summary line fits on one terminal line. Lines wider than the terminal may display imperfectly — known limitation, not in scope.
