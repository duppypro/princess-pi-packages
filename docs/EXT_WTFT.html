Where The F***ing Tokens?! Utility (wtft.ts)

Domain: Cost Audit, Telemetry & Token Accounting

Overview

The WTFT (Where The F***ing Tokens?!) utility provides visual, granular accounting of your session's token consumption and associated monetary cost ($ USD). It parses the active session's append-only .jsonl log, classifies each historical interaction into one of 9 work types, and renders a stacked, color-coded ASCII bar graph over configurable time intervals.

Command Reference

/wtft [options]

Analyze and display the session cost breakdown in a persistent TUI widget below the editor.

Options

  • -i, --interval <size><m|h|d|w> - Group cost data into arbitrary binned intervals of minutes, hours, days, or weeks (e.g., 1m, 7m, 4h, 1d, 2w; default: 1h).
  • -l, --limit <number> - Limit the number of interval bars displayed (default: 10).
  • -w, --width <number> - Set the maximum character width of the TUI widget (default: 240, dynamically scaled down to active terminal columns).
  • -t, --tz, --timezone <zone> - Specify a target display timezone (e.g., America/New_York, Europe/London, Asia/Tokyo) to bypass UTC systems (default: system local).
  • -c, --cumulative - Render running cumulative sums (default behavior).
  • -b, --bucket - Render discrete binned interval cost buckets.
  • --ticks / --no-ticks - Enable or disable the proportional cost scale ticks above the bars (default: true).
  • -p, --pager - Launch a fullscreen, interactive scrollable TUI pager overlay to browse expanded cost history inside your session.
  • -S, --show / -H, --hide - Show (force refresh) or hide the cost audit widget.

/wtft -o, --other

Outputs a cleanly aligned, descending-frequency histogram of the primary executable/command names (e.g., node, npm, cat, mkdir), their execution counts, and cumulative costs during turns classified as "Other". Under the Pi TUI Widget, this prints the histogram directly into the prompt-response message scroll (via chat notification) without modifying the timeline widget. Under the CLI binary, it appends the histogram to stdout directly below the printed timeline.

wtft [options] (Standalone Shell Binary — Cross-Harness)

Run the cost auditor directly from your host shell outside of Pi. Just like bin/merge.mjs, it is shipped as plain ESM JavaScript (bin/wtft.mjs) so it can be installed globally for Claude Code (npm install -g github:duppypro/princess-pi-packages) without triggering Node's type-stripping errors. You can invoke it as the npm-global wtft, via the ./wtft wrapper from the repo root, or through Claude Code's ! prefix (e.g. !wtft -w 90 -i 1d). It reuses the exact same classification engine as the Pi extension, but outputs directly to stdout, allowing you to pipe the output (e.g., wtft | less -R).

Classification Taxonomy & Priority Rules

To avoid guessing or misrepresenting untraced exchanges, interactions are strictly classified based on the file paths written or read, or the commands executed, under a strict priority system:

1. Writes take priority over Reads

  • If two or more distinct categories are written, the turn is mixed.
  • If exactly one category is written, the turn is classified as that category (spec, code, tests, or research).

2. Reads Fallback

  • If no writes exist, the same logic applies: multiple categories read becomes mixed, while a single category read becomes that category.

3. Command Analysis Fallback

  • If no files are accessed, but a command is executed in bash:
    • If starting with cat, head, or tail -> extracts file path and dynamically maps to Read Fallback (or Write Fallback if using a heredoc cat << 'EOF' > file redirection).
    • If starting with git -> git.
    • If starting with grep, rg, ripgrep, or find -> grep.
    • Any other commands -> other.
  • If no commands exist, but dialogue text is present -> prompt.
  • Otherwise -> other.
Classification Color Code Trigger Criteria
Spec Work (spec) ANSI Bright Green (\x1b[92m) Touches files under docs/ or matching AGENTS.md, ARCHITECTURE.md, README.md, or contains spec planning keywords. Treats any .md file as spec.
Code Work (code) ANSI Orange (\x1b[38;5;208m) Touches files under .pi/extensions/, extensions/, src/, public/, bin/, or debug/. Treats code and log/data files (e.g., .ts, .js, .jsonl, .json, .sql, .txt, and bare files without extensions) and ignores (e.g., .gitignore, .dockerignore) as code.
Tests Work (tests) ANSI Yellow (\x1b[93m) Touches files under tests/ (the dedicated test suite directory).
Research Work (research) ANSI Magenta (\x1b[95m) Touches files under research/ (prototypes and algorithmic experiments).
Git Work (git) ANSI Cyan (\x1b[96m) Runs git bash commands.
Grep Work (grep) ANSI Blue (\x1b[94m) Runs search tools like grep, rg, ripgrep, or find.
Prompt Work (prompt) ANSI Dim White (\x1b[37m) Conversational prompt and response with no files or commands accessed.
Other Work (other) ANSI Dark Grey (\x1b[90m) Generic commands, default fallback.

TUI Visualization Style

The TUI offers two distinct high-fidelity visualization styles depending on the display mode:

  • Cumulative Mode (Stacked Bars): Renders a stacked, horizontal binned bar representing overall running cost over time. Each segment is allocated its proportional share of block characters (█, ▒, ░) using a generalized implementation of the Largest Remainder Method, ensuring perfect pixel alignment.
  • Bucket Mode (Scatter/Multi-Line Plot): Acts as a non-stacked point-of-spend multi-line chart. Each category is evaluated independently, placing exactly one colored marker block at its absolute visual position on the scale. Categories that did not spend anything in that interval are omitted. To resolve overlap conflicts where multiple categories happen to fall onto the same horizontal index, characters are plotted in the reverse priority of the legend (from right to left, i.e., Other -> Spec) ensuring that more critical categories like Spec always overlay and sit on top of others.

Legend indicators are placed dynamically on the top title row (or on the second row if the terminal width is narrow), completely removing the redundant 'Legend:' prefix as it is self-evident.

To maximize space and minimize vertical height, the ticks scale and dollar labels are collapsed into a single, ultra-clean horizontal row (when ticks are enabled). The dollar labels themselves (e.g. $0.00, $scaleMax/2, and $scaleMax) are embedded directly inside the horizontal box-drawing line and styled as inverted blocks. To ensure the date indicator on this top ticks row aligns perfectly with the rest of the visual hierarchy, it is styled identically to the standard day-change separator rows (Dark Grey/Bright Black) and rendered as ─── Mmm-DD ─── leading directly into the dollar scale line.

Architectural Design

1. Log Parser: A persistent background daemon (wtft-daemon) watches the session .jsonl file, classifies each interaction, and writes pre-computed entries to a harness-agnostic tag file (wtft-tags/*.tag.v2.2.0.jsonl). In Pi, the widget reads directly from ctx.sessionManager.getEntries() (in-memory) but also spawns a log parser to keep the tag file warm for CLI use.

2. Binning Engine: Groups entries by their timestamp into chronological bins based on the requested interval (e.g., 6-minute windows for granular auditing, or 1-day windows for multi-day tracks).

3. Scaling & Layout: Normalizes the absolute length of each interval's ASCII bar based on the maximum cost in any single interval (typically capped at 30-40 characters wide to fit standard terminal outputs gracefully). Each sub-segment within that bar is allocated its proportional share of characters based on its absolute cost contribution, without rendering percentages or additional trailing text.

4. Recursive Subagent Rollup: When a Claude Code session log is targeted, WTFT auto-detects if a subagent directory exists (e.g., <session-id>/subagents/). If present, WTFT recursively reads all subagent .jsonl logs and blends their interactions chronologically based on their individual timestamps. This guarantees that binned timelines and other command histograms represent the true, cumulative cost and work allocations of the entire multi-agent session tree.

5. Smart Session Auto-Discovery & Selector: When running WTFT without an explicit session path option, WTFT discovers all available session logs for the current repository and environment. Sessions are identified by their parent directory slug (e.g. --home-<user>-git-projects-<project>-- for Pi, -home-<user>-git-projects-<project> for Claude). The selector display shortens the slug to a compact form: ~/g-p/<project>/<date>...<uuid-tail> where g-p is the first-letter shortening of git-projects and the project name is kept in full. UUIDs are truncated to the last 4 hex characters. Session age is shown as a relative time (e.g. 2h ago, 3d ago). If multiple candidates exist:

  • Interactive Selector (TTY): If standard output is a real terminal (interactive TTY), WTFT renders an interactive selection screen showing the compact path slug, estimated cost, turn count, harness type, and relative time since last active.
  • Non-Interactive List Fallback (Pipes/Claude Code '!'): If standard output is non-interactive (such as running !wtft inside Claude Code where standard input is closed), WTFT defaults to the newest session and prints a clean, numbered list of the top candidates. Users/agents can target any older session easily by running wtft -s <number>.

Log Parser Health Monitoring

The widget title line shows a colored status indicator for the background log parser:

  • 🟢 live — log parser is running and processing new interactions
  • 🔴 stopped HH:MM — log parser exited (idle timeout or crash), shows last heartbeat time
  • 🟡 restarting... — log parser is being restarted

In CLI --watch mode, press r to restart the log parser. The footer shows r to restart log parser in red when the parser is stopped. The Pi widget auto-spawns a log parser on session start to keep the tag file warm for CLI use.

Management commands: --list (list running parsers), --cleanup (kill stale parsers), --restart (restart all), --stop <session> (stop one).
