Part 1: Critique of Past Refactoring (The /serve Analysis)

In our previous review of the /serve refactor, we evaluated the breakdown of a massive monolithic command handler. The associated report (SERVE_REFACTOR_ANALYSIS.html) relied heavily on static analysis heuristics (Cyclomatic Complexity, Halstead metrics). Here is a frank developer-centric assessment of that refactoring effort.

The Good:
1. Identifying the monolith: Recognizing that a 265-line if/else chain is unmaintainable was correct.
2. Deduplication: Extracting the fallback kill -9 pattern into a shared process helper successfully removed duplicated logic.
3. Dispatch Table Routing: Moving away from sequential if/else checks to a mapped route array reduced branching logic.
The Bad (Where the analysis fell short):
1. Loss of Type Safety: The refactored handlers lazily typed the Pi context as ctx: any. This defeats the purpose of using a typed SDK in TypeScript.
2. Leaky Closures & Global State: The extracted functions remained inside the serveExtension(pi) closure and mutated global variables (like isWidgetVisible) instead of being pure functions. This prevents true modularity and testability.
3. Brittle Argument Parsing: Relying on regex (/^(--kill|-k)/.test(args)) for command routing is a fragile anti-pattern compared to a robust CLI argument parser.
4. Vanity Metrics: The analysis report proudly highlighted a 60% reduction in "Worst Single Unit" complexity. This is a mathematical tautology (chopping a block into six pieces naturally makes each piece smaller). It did not actually decrease the entropy or structural coupling of the code.

Part 2: Current State of Other Tools (TPM, MERGE, WTFT)

Reviewing our existing toolset reveals similar architectural anti-patterns that need addressing before we expand to support new platforms (like Open Code, Codex, or Gemini CLI).

1. Rate Limiter (TPM / rate-limiter.ts)

Observations:

  • Hardcoded Model Taxonomy: The getModelShortName function and MODEL_QUOTA_REGISTRY are heavily hardcoded to specific Claude and Gemini model strings. Adding OpenAI or Codex support here requires modifying the core tool logic.
  • Session Parsing Duplication: It implements its own custom `.jsonl` file reader to scan ~/.claude/projects/ and ~/.pi/agent/sessions/.
  • Tightly Coupled State: Widget rendering, TPM math, and file I/O are interwoven in updateRateLimiterWidget.

2. WTFT (wtft.ts)

Observations:

  • Manual Argument Parsing: Contains a massive 100+ line parseArgs function filled with if/else if string matching for flags like --limit, -H, --tz.
  • Session Parsing Duplication: Just like the Rate Limiter, WTFT parses the same session `.jsonl` files to compute token usage, but does so using entirely different code.

3. Merge (merge.ts)

Observations:

  • Recently refactored to extract core Git logic into lib/merge/core.ts.
  • Successfully implemented the cross-harness `.mjs` CLI port pattern.
  • Missed Opportunity: Still uses custom localized help-manifest loading instead of a centralized help/manifest rendering utility.

Part 3: DRY Opportunities & Cross-Harness Extensibility

To support Claude Code, Pi, and future platforms (Open Code, Codex, Gemini CLI) efficiently, we must abstract away environment-specific logic. We need a core set of shared libraries.

Module Concept Problem Solved Target Consumers
lib/core/args.ts Replaces brittle regex and massive if/else chains with a unified, typed CLI argument parser (similar to minimist/mri). serve, wtft, merge
lib/core/sessions.ts Unifies the logic for locating, reading, and tailing `.jsonl` session files across Pi, Claude Code, and future agent harnesses. rate-limiter, wtft
lib/core/models.ts Centralizes the taxonomy of LLMs. Maps raw model strings (e.g. claude-3-5-sonnet-20241022) to shortcodes, context limits, and token rate quotas. Easily extensible for OpenAI/Codex. rate-limiter, wtft
lib/core/ui.ts Abstracts terminal output. Can render as an ANSI table for standard CLI tools or integrate with Pi's interactive TUI context. Also unifies `--help` manifest loading. All Extensions and CLI bins

Part 4: The June Refactoring Plan

Phase 1: Centralize Argument Parsing & Help Menus
  • Create lib/core/args.ts.
  • Refactor wtft.ts and serve.ts to strip out custom string matching.
  • Unify manifest-driven --help output logic across all commands.
Phase 2: Unify Log Parsing & Model Taxonomies
  • Extract the model token limits and shortcode generation from rate-limiter.ts into lib/core/models.ts. Ensure the registry is agnostic enough to accept new providers.
  • Extract `.jsonl` reading logic from both rate-limiter and wtft into a shared lib/core/sessions.ts.
Phase 3: Cross-Harness UI & CLI Support
  • Port rate-limiter.ts to bin/tpm.mjs so Claude Code users can type !tpm to view their current rate limits without needing the Pi TUI.
  • Ensure all extracted core modules use strict TypeScript interfaces, avoiding the any anti-pattern observed in the /serve refactor.

By implementing this plan, we transition from writing isolated "scripts" to maintaining a cohesive, scalable toolchain capable of running anywhere.
