Git→main Merger Utility (merge.mjs)

Domain: Git Automation, Multi-Worktree and Single-Checkout Workflows & Step-Based Releases

Overview

The /merge command provides a zero-token, local utility to automate the validation and integration of a completed Step 5 feature branch (or a specific stable commit) into the main branch. It ensures compliance with the Princess-Pi 5-Step Development Flow without invoking costly LLM reasoning turns.

CLI Mode (Outside Pi Coding Agent) — Cross-Harness

bin/merge.mjs is a standalone port of the same validation/merge/push sequence (plain execSync git calls, no Pi dependency). It is plain ESM JavaScript (#!/usr/bin/env node) rather than TypeScript on purpose: when installed globally for Claude Code (npm install -g github:duppypro/princess-pi-packages), the bin lives under node_modules/, where Node refuses --experimental-strip-types (ERR_UNSUPPORTED_NODE_MODULES_TYPE_STRIPPING); plain JS needs no tsx and no build step. Invoke it as the npm-global merge [commit-ish], via the ./merge wrapper from the repo root, or through Claude Code's ! prefix (e.g. !merge 638e9b4) — Claude has no extension-dispatch that bypasses the model the way Pi's registerCommand does, so the CLI is how it stays zero-token there. Behavior matches the Pi extension; only the output channel differs (console.log instead of ctx.ui.notify). The typed twin extensions/merge.ts remains the Pi-runtime version.

Worktree vs single-checkout: if a dedicated main git-worktree exists, the merge runs there and never disturbs the feature checkout (Pi's original multi-worktree design). Otherwise — the common Claude Code case — it falls back to an in-place single-checkout merge in the current repo, always returning you to your feature branch afterward and rolling back (git merge --abort) on conflict.

Verification Rules

Before any merge operations occur, the extension strictly enforces the following rules in sequence:

  1. No Self-Merge: Rejects running the command if the active worktree is already on the main branch.
  2. Clean Feature Worktree: Verifies that the current feature worktree contains no uncommitted (staged or unstaged) modifications.
  3. In Sync with Origin: Fetches origin and ensures the targeted commit-ish is present in the history of the remote tracking branch (preventing unpushed commits).
  4. Step 5 Validation: Validates the targeted commit's subject against /^Code and Spec Approved(\s*\([^)]*\))?\s*:/ — i.e. it must begin with Code and Spec Approved followed by an optional parenthetical (e.g. Code and Spec Approved (Step 5):) and a colon, conforming to Step 5 of the Princess-Pi standard. If it does not, the command fails and automatically suggests the most recent historical commit that does meet this standard.
  5. Clean Target (when a main worktree exists): If a dedicated main worktree is found, verifies it contains no uncommitted changes. In single-checkout mode this step is skipped — the current feature worktree was already verified clean above, and the tool switches branches in place.

Merge and Push Execution

Once all validations pass, the utility integrates and pushes automatically. Two paths:

Multi-worktree path — inside the dedicated main worktree directory:

  1. Switches/ensures the main worktree is on the main branch.
  2. Pulls the latest updates from origin/main (--ff-only).
  3. Merges the validated target commit into main (leaving any subsequent unapproved commits on the feature branch).
  4. Pushes the newly merged main branch to origin.
  5. Logs success; the active feature worktree was never touched.

Single-checkout fallback — in the current repo, wrapped so it always returns you to the feature branch:

  1. Checks out main in place (creating it from origin/main if the local branch is absent) and pulls --ff-only.
  2. Merges the validated target commit into main.
  3. Pushes main to origin.
  4. On any failure, runs git merge --abort and reports an LLM-actionable error; a finally step always checks the feature branch back out, so you are never stranded on main or mid-merge.
