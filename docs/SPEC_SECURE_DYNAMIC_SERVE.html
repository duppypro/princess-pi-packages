Spec: Secure Dynamic /serve Tool with DNS, trusted SSL, and Google OAuth

This specification outlines the architecture, routing patterns, ACL security, and local tool changes required to securely serve local web assets under https://princess-pi.dev/preview/<client_slug>/ paths.

1. Make vs Buy Analysis

Solution OAuth & Gating Zero SSL Overhead Locally Multi-Tenant Map Gating Integration with Local CLI Dynamic Path-Based Routing
ngrok ⚠️ Paid/Complex ✅ Yes ❌ No ❌ Custom wrapper needed ❌ Random domains
Cloudflare Tunnels ⚠️ Needs Access policy ✅ Yes ❌ No ❌ Heavy agent ⚠️ Subdomains only
/serve Upgrade (Map Pattern) ✅ Yes (Unified port 4182) ✅ Yes (Loopback only) ✅ Yes (Map ACLs) ✅ Yes (Zero config) ✅ Yes (Exact subpaths)

Conclusion: Make (Upgrade /serve). Upgrading our existing tool with a VPS-side NGINX map-based reverse proxy configuration offers complete end-to-end routing control, Zero-SSL overhead locally, and a unified Google login without requiring extra runtime dependencies or third-party service subscriptions.

2. Dynamic Client Slug Creation Rules

To support clean, multi-tenant workspace routing, the local /serve command will automatically construct the <client_slug> using the following fallback rules:

  1. Inside a Git Repository (Subdirectory): If serving a subdirectory (e.g., docs) inside a Git repository (e.g., rogue-savvy), the slug is: <repo_name>/<dir_name> (e.g., rogue-savvy/docs).
  2. Inside a Git Repository (Root/No Directory Specified): If serving the root directory of a repository, the slug is just the repository name: <repo_name> (e.g., rogue-savvy). No /root or trailing slash is added.
  3. Outside a Git Repository (Fallback): If run outside a Git repository, the slug is determined by the current directory's folder name.

B. Dynamic NGINX Map Routing

NGINX maps act as a secure, fast, and light database. Both maps reside in /etc/nginx/ and are writable by the princess-pi group to allow the `/serve` CLI tool to write port mappings without requiring sudo root privileges.

1. Access Control List Map: /etc/nginx/serve-acls.map

# Matches authorized Google emails to their allowed client slug.
# "all" is a special value granting internal administrators global access.
david@princess-pi.dev        "all";
princess@princess-pi.dev      "all";

# Client specific mappings
client-a@gmail.com            "client-a";
client-b@hotmail.com          "client-b";
rogue-client@gmail.com        "rogue-savvy";
rogue-docs-client@gmail.com   "rogue-savvy/docs";

2. Port Routing Map: /etc/nginx/serve-ports.map

# Dynamic loopback ports updated in real-time by the local `/serve` CLI.
rogue-savvy                  8080;
rogue-savvy/docs             8081;

C. NGINX Site Block Configuration

Injected at the top level of /etc/nginx/sites-available/princess-pi.dev (outside the server block):

# --- Maps email address to allowed client slug ---
map $user_email $allowed_client_slug {
    default "";
    include /etc/nginx/serve-acls.map;
}

# --- Maps client slug to active loopback port ---
map $client_slug $active_serve_port {
    default "";
    include /etc/nginx/serve-ports.map;
}

Inside the HTTPS server block:

# --- oauth2-proxy endpoints for princess-pi.dev (Port 4182) ---
location /oauth2/ {
    proxy_pass       http://127.0.0.1:4182;
    proxy_set_header Host                    $host;
    proxy_set_header X-Real-IP               $remote_addr;
    proxy_set_header X-Forwarded-For         $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto       $scheme;
    proxy_set_header X-Auth-Request-Redirect $request_uri;
}

# --- oauth2 auth check subrequest ---
location = /oauth2/auth {
    proxy_pass       http://127.0.0.1:4182;
    proxy_set_header Host             $host;
    proxy_set_header X-Real-IP        $remote_addr;
    proxy_set_header X-Forwarded-For  $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header Content-Length   "";
    proxy_pass_request_body off;
}

# --- Dynamic Client Preview /preview/<client_slug>/ ---
# Strictly allows optional nested paths (e.g. repo/dir) while blocking parent traversal.
location ~ ^/preview/(?[a-zA-Z0-9_-]+(?:/[a-zA-Z0-9_-]+)?)(?/.*)?$ {
    # 1. Trigger Authentication
    auth_request /oauth2/auth;
    error_page 401 = /oauth2/start?rd=$request_uri;

    # Capture the email returned by oauth2-proxy
    auth_request_set $user_email $upstream_http_x_auth_request_email;

    # 2. Authorization Rules
    if ($allowed_client_slug = "") {
        return 403 "Forbidden: Email not authorized to view previews.";
    }

    set $auth_status "deny";
    if ($allowed_client_slug = "all") {
        set $auth_status "allow";
    }
    if ($allowed_client_slug = $client_slug) {
        set $auth_status "allow";
    }

    if ($auth_status = "deny") {
        return 403 "Forbidden: You do not have permission to view this client's preview.";
    }

    # 3. Port Lookup
    if ($active_serve_port = "") {
        return 503 "Service Unavailable: Preview server is not currently running.";
    }

    # 4. Proxy-Pass to Dynamic localhost port
    proxy_pass http://127.0.0.1:$active_serve_port$remaining_path$is_args$args;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    # WebSocket / SSE support (critical for hot reloading!)
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_buffering off;
}

# --- Root location fallback for absolute/root-relative assets ---
location / {
    # If the referer indicates the request originated from a client preview,
    # route the asset to the correct client port.
    if ($http_referer ~* /preview/(?[a-zA-Z0-9_-]+(?:/[a-zA-Z0-9_-]+)?)/) {
        set $client_slug $referer_slug;
        proxy_pass http://127.0.0.1:$active_serve_port;
    }

    # Default to main portfolio/backend app
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

4. Tool Updates (/serve Command)

  1. No SSL Generation: Remove SSL/TLS file generation and the reliance on ~/.pi-certs inside the local tool runtime. All local servers run strictly on plain HTTP.
  2. Relative EventSource Connection: In extensions/lib/serve/run-live-server.js, modify the browser injection script to connect to a relative EventSource("__live-reload") instead of the root-relative /__live-reload.
  3. Live Port Writing: When a server is successfully spawned or killed, the tool directly updates the key-value entry in /etc/nginx/serve-ports.map:
    • Start: Writes <client_slug> <port>;.
    • Stop: Deletes or comments out the entry for the matching slug.
  4. Public URL Output: Replaces local loopback display outputs with the clean URL:
    🚀 Served at: https://princess-pi.dev/preview/<client_slug>/.

5. Verification Plan

  • Local Command Tests: Run /serve docs inside rogue-savvy and verify it starts an HTTP server on loopback, correctly identifies slug rogue-savvy/docs, writes to /etc/nginx/serve-ports.map, and prints the public URL.
  • NGINX Compilation Tests: Validate NGINX configuration using sudo nginx -t to verify mapping directives and dynamic regex patterns compile without error.
  • Access Gating Verification:
    • Authorized user matching slug (e.g., rogue-client@gmail.com accessing /preview/rogue-savvy): HTTP 200.
    • Authorized user with different slug (e.g., client-a@gmail.com accessing /preview/rogue-savvy): HTTP 403 Forbidden.
    • Unmapped user (e.g., stranger@gmail.com): HTTP 403 Forbidden.
    • Company administrator (e.g., david@princess-pi.dev): HTTP 200 for all paths.
  • Live Reload & Asset Resolving Verification: Load a page via the preview URL, check that assets (images, CSS, JS) and EventSource hot-reload connection resolve correctly.
