Pi Extensions: Architecture Manifesto

This document serves as the high-level overview for Princess-Pi's agent extensions. As we iterate, the architectural philosophy defined here remains constant, while individual extension sub-specs may evolve independently.

Light Domain Driven Design (DDD) Strategy

Pi extensions are structured using a lightweight Domain-Driven Design approach to prevent "God Modules" and ensure code reusability. Extensions must be cleanly decoupled into the following layers:

  • Application Layer: The orchestrator. Parses CLI commands (e.g., /serve), routes intents, and registers capabilities with Pi.
  • Domain Layer: Business logic. Defines core entities (e.g., ServerInstance) and determines rules like bounding contexts to Git repositories.
  • Infrastructure Layer: System boundaries. Handles OS-level execution (ps aux), file system access, network requests, and Pi session persistence logs.
  • Presentation Layer: Formatting. Pure functions that handle ANSI escape sequences, padding, and UI widget string construction without mutating state.
